# Evals Specialist Knowledge Bank

**Target models:** Promising OSS such as qwen/qwen3.5-9b (smoke leader) and peers from full matrix. Loaded via LM Studio OpenAI-compatible endpoint (http://localhost:1234/v1 by default).

**Role:** Specialist in the local OSS model evaluation harness. Focus: daily briefs, hourly interest tracking, tool-assisted research/plan for model placement decisions, token-efficient operation on constrained hosts (e.g. 8 GB VRAM).

## Harness Ground Truth (live repo source of truth — always re-verify with rg, pnpm capture:system, registry export, results/*)
- **No paid inference in automated runs.** LM Studio + lms CLI only for matrix, local eval runner, agent traces.
- **Core surfaces:** registry/ (models.json from lms ls --json + load-profiles.json), baselines/ (imported only), suites/promptfoo/ (matrix YAML + prompts + sandboxed tool callbacks), suites/deepeval/ (agent trace metrics), scripts/ (orchestration, matrix, baseline import/compare, no product runtime).
- **Lanes:** research, plan, build, tool-call. Every model sees identical tasks; only provider + load profile (gpu_full / gpu_offload) differs.
- **Load profiles (see registry/load-profiles.json + results/system-profile.json):** gpu_full uses `lms load --gpu max`; gpu_offload uses `lms load --gpu off`. Large models and some 9B benefit from offload on this hardware per measured data.
- **Results layer (primary, no Docker required):** results/matrix-*.jsonl, matrix-summary.json (leaderboard + pass rates + durationMs), optimization-state.json (checkpoint, userGoal flags, placement hints), baseline-comparison.jsonl, user-judge-queue.jsonl. JSONL + reports are durable.
- **Baselines:** Never generated by harness. Use `pnpm baseline:prompts`, collect outside (subs or `pnpm baseline:collect` with Vertex/Azure credits), then `pnpm baseline:import`. Manifest in baselines/manifest.json owns taskId/lane/promptFile mapping.
- **Tool calling in evals:** Strictly sandboxed. Promptfoo uses `functionToolCallbacks` (JS functions returning static/mocked data, no fs/shell/net). DeepEval traces use controlled tool lists. Real side-effects only in host scripts (this tracker) with timeouts and safe paths.
- **Agent metrics (DeepEval):** TaskCompletionMetric, ToolCorrectnessMetric, PlanAdherenceMetric on multi-step traces. Align with Promptfoo cases.
- **Token / speed recording:** run-local-eval and matrix capture latencyMs + OpenAI usage (prompt_tokens, completion_tokens). Matrix cells record durationMs. Specialist runs (tracker) must explicitly compute and record tps (completion_tokens / seconds) + full usage.
- **Judge / compare:** Primary is user review of local vs imported baseline (via queue). Optional Azure/Vertex rubric env-gated only. Compare via pnpm compare:baseline.
- **Verification ladder (per AGENTS.md):** pnpm capture:system, pnpm registry:export, pnpm baseline:prompts, pnpm matrix:smoke, pnpm compare:baseline, pnpm judge:queue, pnpm eval:deepeval, pnpm verify, git diff --check. Use rg for all searches.
- **Boundaries (strict):** This repo evaluates models/workflows. Does **not** implement Harness, Hermes, Zavi, or Studio product runtimes/orchestrators. Tracker script is harness-owned support for W7 specialist use-case only.

## Daily Briefs + Hourly Web Search Interest Tracker
- **Goal:** Unload daily/interest tracking, local chat, planning, and tool calls (gh, fs read, web via free libs) to local models. Produce actionable briefs that inform harness ops (model selection, tuning targets, 3-solid-models criteria).
- **Typical interest queries:** local OSS evals progress, qwen3.5-9b / gemma variants real-world perf on low VRAM, github activity on the evals transparency repo, recent OSS releases/tools for evals (free only: no paid search).
- **End-to-end flow (baby steps to full):**
  1. Load specialist KB + recent summaries (matrix-summary, optimization-state, system-profile) into system prompt.
  2. Use tools in ReAct-style loop (think, select tool+args, call, observe, repeat <= max_steps, then finalize).
  3. Tools (real in tracker, sandboxed in suites):
     - web_search(query): duckduckgo-search (free, no key).
     - fetch_page(url): trafilatura (clean extract, no paywalls abuse).
     - read_file(relative_path): safe repo read (whitelist docs/, registry/, suites/, scripts/, results/ allowed summaries; block .. and sensitive).
     - github via gh CLI: `gh repo view`, `gh search`, `gh issue list`, `gh pr list` for public repo signals (JamiStudio/local-evals).
  4. Structured brief output (markdown):
     - Header with timestamp, model, load hint, specialist mode.
     - Web / search signals (with sources).
     - GH / repo activity.
     - Harness notes (placement from live results, token considerations).
     - Recommended actions (plan steps, 1-5 concrete).
     - Token/speed summary (prompt+completion, tps, wall time).
  5. Persist: results/daily-briefs/brief-<iso>.json (full record with steps, usage, tps). (results/ artifacts selectively committed per .gitignore.)
- **Quality criteria (for evals vs baseline):** factual + sourced, concise, harness-aware (references lanes/profiles/results), actionable plan, demonstrates tool correctness + plan adherence, reasonable token use.
- **On-demand vs loop:** Default single run (on-demand). `--loop` for recurring (real use; sleeps 1h; use for host cron). Support `--dry-run` for smoke verification (simulated tokens/speed, no LM or net calls).
- **Token speed:** Always measured and recorded for specialist runs. Compare across models/presets in future matrix extensions.

## Simple Tool-Calling Agent Loop (ReAct-style)
- Thin loop owned by tracker (reusable pattern, not product orchestrator):
  - Inject tools schema (OpenAI function calling format) + specialist system.
  - Loop: model call (with tools) → if tool_calls: execute real (or mocked) tool → append observation as tool message → continue; else treat content as final answer.
  - Cap steps, log every decision/obs/usage for metrics.
  - On final: extract brief, compute tps from usage + timing, persist + print.
- In suites: the same loop pattern is **evaluated** (not implemented) via Promptfoo tool cases (sandbox callbacks) + DeepEval trace metrics (tool correctness, plan adherence).
- Example loop skeleton lives in scripts/daily-brief-tracker.py (search for `def run_brief_agent`).

## Specialist vs General Usage
- **Specialist mode** (`--use-specialist`): Load this KB + live results summaries. Produces briefs tuned to harness (e.g. "qwen3.5-9b @ gpu_offload preferred per O3; use for daily research/plan supporting role on 8GB").
- **General:** Plain model call without KB — weaker on harness-specific recs, token awareness, tool discipline for this domain.
- Eval additions (W7): Promptfoo test cases for briefs-quality research + tool-use tracker flows; DeepEval tests exercising ToolCorrectnessMetric + PlanAdherenceMetric + briefs task vs reference/baseline. Measure quality, tool accuracy, plan steps, token speed.
